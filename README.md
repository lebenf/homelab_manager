# homelab_manager

Tool repository for a file-based, declarative, Podman-first (Docker-compatible)
self-hosted homelab. This is **not** a repository you deploy directly — it is
the reusable installer, parametrized templates, shared scripts and canonical
`CLAUDE.md` used to generate and keep in sync one *instance repository* per
server.

## Two-repository model

- **Tool repo** (this repository): `install.sh`, `templates/*.tmpl`,
  `templates/app-scripts/`, `scripts/`, and the canonical `CLAUDE.md`.
  Improvements made here (a better script, a fixed rule in `CLAUDE.md`, a
  template tweak) benefit every server the next time it's synced.
- **Instance repo** (one per server, generated by `install.sh`, never built
  by hand): contains that server's `STATE.md`, its rendered `infra/`
  (`core-compose.yml`, `traefik/traefik.yml`, `scripts/`,
  `templates/app-scripts/`), `apps/`, and `infra/.env` (not committed).
  Claude Code opens and edits the *instance* repo when it later adds or
  changes an app — never this tool repo.

See `WORKFLOW.md` for the full set of fixed conventions (placeholder tokens,
network/service naming, container-engine resolution, etc.) shared by every
file in this repository.

## Usage

Run once per server to create its instance repo:

```
./install.sh --target-dir <path> --runtime podman|docker \
  [--base-domain <domain>] [--acme-email <email>] [--server-name <name>] \
  [--backup-dest <dest>] [--non-interactive]
```

Any of `--base-domain`, `--acme-email`, `--server-name`, `--backup-dest` left
unset are prompted for interactively, unless `--non-interactive` is given —
in which case `--base-domain`, `--acme-email` and `--server-name` become
required flags (`--backup-dest` stays optional; leave it out to configure a
backup destination later).

`install.sh` never starts a container. When it finishes it prints the manual
next steps:

1. Review the generated `infra/.env` in the instance repo.
2. If using Podman rootless, apply the printed prerequisites: the privileged
   port `sysctl` (`net.ipv4.ip_unprivileged_port_start`) and
   `loginctl enable-linger <user>`.
3. Start the core services: `cd <target-dir>/infra && <runtime> compose -f core-compose.yml up -d`.

### Refreshing an existing instance: `--update-core`

Re-run `install.sh` with `--update-core` against an already-generated
instance directory to resync `CLAUDE.md`, `infra/scripts/` and
`infra/templates/app-scripts/` from this tool repo. It never touches that
server's `STATE.md`, `infra/core-compose.yml`, `infra/traefik/traefik.yml`,
`infra/.env`, or `apps/`.

```
./install.sh --target-dir <path> --runtime podman|docker --update-core --non-interactive
```

## Adding a new app to an instance

Work from inside the instance repo, not this tool repo. Follow that
instance's own `CLAUDE.md` (copied verbatim from here) and use
`infra/scripts/app.sh <app-name> <action>` to run the lifecycle scripts
(`install`, `update`, `backup-now`, `remove`, `start`, `stop`, `restart`,
`status`, `logs`) once an app has been added.

## Validating this tool repo

```
bash tests/validate.sh
```

Syntax-checks every shell script and validates every compose/YAML file
(including the `.tmpl` templates) in this repository. `tests/test-install-e2e.sh`
additionally exercises `install.sh` end to end against a disposable temporary
directory.
